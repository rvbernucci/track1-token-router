# Gemma E2B Token Ladder

Updated: 2026-07-10

## Question

Determine whether extra local decode budget converts previously incorrect Gemma E2B answers into correct answers, and whether FunctionGemma's intent plus five scores predict that recovery well enough to route safely.

## Controlled Design

1. Generate every calibration answer at 96 tokens.
2. Obtain independent judgments from pinned Minimax M3 and Kimi K2.7 Code teachers.
3. Accept a local answer as correct only when both teachers return `correct`.
4. Retry only non-unanimous-correct tasks at 192 tokens.
5. Repeat independent judgments on the new candidate IDs.
6. Retry the remaining tasks at 384 tokens.
7. Correlate first successful limit with intent, five FunctionGemma scores, output length and latency.

Each observation has a distinct `candidate_id`, original `task_id` and `generation_limit_tokens`. Teacher disagreements remain unresolved evidence and never become positive labels automatically.

## Quantization Baseline

The pinned `gemma-4-E2B-it.litertlm` artifact is already an official quantization-aware trained package with mixed 2-bit, 4-bit and 8-bit weights, memory-mapped embeddings and demand-loaded multimodal components. The experiment is not running FP16.

Do not apply an additional post-training quantization without an A/B quality and runtime benchmark. Smaller weights can reduce memory bandwidth, but unsupported kernels, dequantization overhead and quality loss can erase the gain.

## Speed Challengers

- official speculative decoding on the same standard artifact;
- the official 2.0 GB Web text-only artifact, only if it proves compatible with native Linux `amd64` LiteRT-LM;
- dynamic token ceilings selected from the empirical recovery matrix.

Every challenger must report exact artifact hash, peak RSS, p50/p95 latency, task-level correctness and combined 4 GB behavior. The final selection optimizes accuracy under the ten-minute wall-clock constraint, not model size alone.

## Stage 96 Evidence

- tasks: `93`;
- unanimous correct: `15`;
- unanimous incorrect: `60`;
- teacher disagreement: `18`;
- observed latency: p50 `5.51 s`, p95 `8.96 s`, quarantined because the LiteRT OpenAI server selected its default AMD GPU backend despite the client process being pinned to two CPU cores;
- independent judge spend: Minimax approximately `$0.02`; Kimi `$0.2413914` recorded, excluding one malformed capped attempt.

Mean FunctionGemma scores for unanimous-correct cases were lower than unanimous-incorrect cases in format complexity (`3.67` vs `5.65`), generation demand (`3.47` vs `4.80`) and reasoning demand (`3.80` vs `4.97`). These are descriptive signals only; promotion requires held-out calibration.

## Ladder Result

| First unanimous-correct ceiling | Newly recovered | Cumulative correct |
|---|---:|---:|
| 96 | 15 | 15 |
| 192 | 10 | 25 |
| 384 | 8 | 33 |
| still not unanimous-correct | 60 | 33 |

The 384-token recoveries were limited to NER, code debugging and math reasoning. This is not sufficient evidence for a general 384-token local route. All latency columns produced by the initial ladder are GPU-backed diagnostic measurements and must not be used for grader runtime claims.

### Recovery versus judge variance

The task-level prefix audit refined the raw approval counts:

| Transition | Retried | Genuine prefix-growing recovery | Identical answer, changed judgment |
|---|---:|---:|---:|
| 96 -> 192 | 78 | 7 | 3 |
| 192 -> 384 | 68 | 8 | 0 |

The three identical 96/192 answers are judge variance, not evidence that the larger token ceiling helped. They remain in the immutable raw consensus evidence, but are excluded from the marginal token-benefit target used by the router. A positive token-budget label now requires both a strict prefix-growing continuation and unanimous correctness at the higher ceiling.

The reproducible analysis and task-level classifications are generated by `scripts/analyze_e2b_token_ladder.py` into `token-ceiling-frontier.json` and `token-ceiling-frontier.md`.

Conditional recovery by family:

| Intent | 96 correct / tested | 192 recovered / retried | 384 recovered / retried |
|---|---:|---:|---:|
| Code debugging | 1 / 9 | 1 / 8 | 2 / 7 |
| Code generation | 1 / 17 | 1 / 16 | 0 / 15 |
| Factual Q&A | 2 / 7 | 0 / 5 | 0 / 5 |
| Logic puzzle | 3 / 15 | 2 / 12 | 0 / 10 |
| Math reasoning | 1 / 14 | 1 / 13 | 3 / 12 |
| NER | 2 / 14 | 1 / 12 | 3 / 11 |
| Sentiment | 4 / 7 | 1 / 3 | 0 / 2 |
| Summarization | 1 / 10 | 3 / 9 | 0 / 6 |

These denominators are conditional: each higher stage contains only tasks that were not unanimously correct at the previous stage. They support family-conditioned ceilings, not raw cross-family accuracy comparisons.

The corrected runtime uses `scripts/litert_cpu_server.py`, which forces `backend=cpu`, two threads and an explicit KV context. A 1024-token context failed long benchmark prompts; the controlled CPU comparison therefore uses 2048 tokens and must pass the final 4 GB Docker gate.

## Corrected CPU Evidence

The standard 2.6 GB artifact completed all 93 tasks on the forced two-thread CPU server with context 2048 and a 96-token ceiling:

- failures: `0/93`;
- p50 latency: `4.22 s`;
- p95 latency: `7.09 s`;
- maximum latency: `7.29 s`;
- outputs identical to the initial standard-artifact GPU run: `93/93`.

The full 96-token pass consumed `393.22 s` of summed model latency. This leaves only about `206.78 s` inside the ten-minute evaluator deadline for FunctionGemma, orchestration, retries and Fireworks calls. Consequently, 192 and 384 tokens cannot be global defaults; they must be selectively purchased from a held-out estimate of marginal recovery probability.

The output identity means the teacher-derived quality labels remain usable, while the GPU latency measurements remain quarantined.

The official 2.0 GB Web text-only artifact is not a CPU replacement. It completed the GPU run quickly but failed `93/93` on the forced CPU backend because it has no native `TF_LITE_PREFILL_DECODE` section. Reject it from the grading container.

Speculative decoding was supported by the standard artifact. On the 12 longest baseline outputs it accelerated 11 tasks and improved mean latency by `1.17x`, but changed 9 answers. Independent teachers produced the same outcome classes as the non-speculative baseline for this sample: 11 incorrect and one disagreement. This proves no quality regression within that negative sample, but supplies no positive-case promotion evidence. Keep speculative decoding experimental until a correct-task holdout and 4 GB RSS test pass.
